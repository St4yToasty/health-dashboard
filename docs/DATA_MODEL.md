# Data model

Schema design for v1. Uses **wide per-metric tables** (one table per metric type), not a generic EAV table. Each new metric type added later = one migration.

## Conventions

- All primary keys are `bigserial` (`bigint generated by default as identity` if we end up on Postgres 17).
- All timestamps are `timestamptz` and stored in UTC. Day-boundary math is done at query time in `Europe/Lisbon`.
- Every metric table has a `source` column with values from a shared `metric_source` enum:
  - `apple_health` — from HAE auto-sync
  - `manual` — entered by the user in the UI
  - `import` — bulk historical import (one-time)
  - `derived` — computed from other rows (e.g. BMI from weight + height)
- **Idempotency:** every metric table has a unique constraint that makes re-ingest safe. Typically `(recorded_at, source)` or `(date, source)` depending on whether the metric is per-instant or per-day.
- Soft-delete is **not** used. Personal data, single user — actual `DELETE` is fine and avoids "is this row real?" ambiguity.

## Enums

```sql
create type metric_source as enum ('apple_health', 'manual', 'import', 'derived');
create type meal_kind as enum ('breakfast', 'lunch', 'dinner', 'snack');
create type workout_kind as enum (
  'walking', 'running', 'cycling', 'strength', 'hiit', 'yoga', 'swimming', 'other'
);
create type sleep_stage as enum ('in_bed', 'asleep', 'core', 'deep', 'rem', 'awake');
create type goal_metric as enum (
  'daily_calories', 'daily_water_ml', 'daily_steps', 'target_weight_kg'
);
```

## Tables

### `body_composition`

One row per measurement. From smart scale (via HAE) or manual fallback.

```sql
create table body_composition (
  id              bigserial primary key,
  recorded_at     timestamptz not null,
  weight_kg       numeric(5,2),          -- nullable; some entries are body-fat only
  body_fat_pct    numeric(4,2),
  muscle_mass_kg  numeric(5,2),
  bmi             numeric(4,2),          -- usually derived, stored if HAE provides
  source          metric_source not null,
  notes           text,
  created_at      timestamptz not null default now(),

  constraint body_composition_at_least_one_value check (
    weight_kg is not null or body_fat_pct is not null or muscle_mass_kg is not null
  ),
  unique (recorded_at, source)
);

create index on body_composition (recorded_at desc);
```

### `body_measurements`

Tape-measure measurements (waist, chest, etc.). Always manual, lower frequency than weight.

```sql
create table body_measurements (
  id            bigserial primary key,
  recorded_at   timestamptz not null,
  waist_cm      numeric(5,1),
  chest_cm      numeric(5,1),
  hips_cm       numeric(5,1),
  thigh_cm      numeric(5,1),
  arm_cm        numeric(5,1),
  neck_cm       numeric(5,1),
  source        metric_source not null default 'manual',
  notes         text,
  created_at    timestamptz not null default now(),

  constraint body_measurements_at_least_one check (
    coalesce(waist_cm, chest_cm, hips_cm, thigh_cm, arm_cm, neck_cm) is not null
  ),
  unique (recorded_at, source)
);
```

### `foods` (personal food library)

User's growing library. Each food is added once and reused.

```sql
create table foods (
  id            bigserial primary key,
  name          text not null,
  brand         text,
  serving_grams numeric(7,2) not null,         -- the canonical serving size
  kcal          numeric(6,1) not null,         -- per `serving_grams`
  protein_g     numeric(5,1) not null,         -- per `serving_grams`
  is_archived   boolean not null default false,
  created_at    timestamptz not null default now(),
  updated_at    timestamptz not null default now(),

  unique (name, brand)
);

create index on foods (lower(name));
```

### `meals` + `meal_items`

A meal is a logical container for a set of food items eaten in the same sitting.

```sql
create table meals (
  id           bigserial primary key,
  eaten_at     timestamptz not null,
  kind         meal_kind not null,
  notes        text,
  created_at   timestamptz not null default now()
);

create index on meals (eaten_at desc);

create table meal_items (
  id          bigserial primary key,
  meal_id     bigint not null references meals(id) on delete cascade,
  food_id     bigint not null references foods(id) on delete restrict,
  grams       numeric(7,2) not null,
  -- Snapshot the macros at log time so they don't shift if the food row is edited later
  kcal        numeric(6,1) not null,
  protein_g   numeric(5,1) not null,
  created_at  timestamptz not null default now()
);

create index on meal_items (meal_id);
```

**Daily totals view:**

```sql
create view daily_nutrition as
select
  date_trunc('day', eaten_at at time zone 'Europe/Lisbon')::date as day,
  sum(mi.kcal)::numeric(7,1) as kcal,
  sum(mi.protein_g)::numeric(6,1) as protein_g,
  count(distinct m.id) as meal_count
from meals m
join meal_items mi on mi.meal_id = m.id
group by 1;
```

### `water_intake`

Each row is one logged drink. One-tap quick-add is the primary entry path.

```sql
create table water_intake (
  id           bigserial primary key,
  recorded_at  timestamptz not null,
  amount_ml    integer not null check (amount_ml > 0 and amount_ml <= 5000),
  source       metric_source not null default 'manual',
  created_at   timestamptz not null default now()
);

create index on water_intake (recorded_at desc);
```

Note: no unique constraint. Drinking water twice in the same minute should produce two rows. HAE's water data (if any) is treated as additive, not authoritative.

### `daily_activity`

Daily aggregates from HAE: steps, distance, active and basal energy. One row per day per source.

```sql
create table daily_activity (
  id                 bigserial primary key,
  day                date not null,
  steps              integer,
  distance_m         integer,
  active_energy_kcal integer,
  basal_energy_kcal  integer,
  flights_climbed    integer,
  source             metric_source not null,
  updated_at         timestamptz not null default now(),

  unique (day, source)
);

create index on daily_activity (day desc);
```

Re-pushes for the same day overwrite (HAE keeps recalculating "today" until the day rolls over).

### `workouts`

Discrete workout sessions from HAE.

```sql
create table workouts (
  id                 bigserial primary key,
  started_at         timestamptz not null,
  ended_at           timestamptz not null,
  kind               workout_kind not null,
  active_energy_kcal integer,
  distance_m         integer,
  avg_hr             integer,
  max_hr             integer,
  notes              text,
  source             metric_source not null,
  created_at         timestamptz not null default now(),

  check (ended_at > started_at),
  unique (started_at, kind, source)
);

create index on workouts (started_at desc);
```

### `sleep_sessions`

One row per sleep session (typically one per night, sometimes naps).

```sql
create table sleep_sessions (
  id            bigserial primary key,
  started_at    timestamptz not null,
  ended_at      timestamptz not null,
  in_bed_min    integer,
  asleep_min    integer,
  deep_min      integer,
  rem_min       integer,
  core_min      integer,
  awake_min     integer,
  source        metric_source not null,
  created_at    timestamptz not null default now(),

  check (ended_at > started_at),
  unique (started_at, source)
);

create index on sleep_sessions (started_at desc);
```

Stage-level columns are nullable because Huawei Band 9 + Sleep Cycle may not always expose them.

### `goals`

First-class. Each row is a (metric, target, period) tuple effective for a window in time.

```sql
create table goals (
  id              bigserial primary key,
  metric          goal_metric not null,
  target_value    numeric(8,2) not null,
  target_date     date,                          -- for end-date goals like target weight
  effective_from  date not null,
  effective_to    date,                          -- null = currently active
  notes           text,
  created_at      timestamptz not null default now(),

  check (effective_to is null or effective_to > effective_from),
  exclude using gist (metric with =, daterange(effective_from, coalesce(effective_to, 'infinity')) with &&)
);
```

The `exclude` constraint enforces non-overlapping windows per metric. Requires `btree_gist`:

```sql
create extension if not exists btree_gist;
```

**Currently-active-goal lookup:**

```sql
create view goals_active as
select distinct on (metric)
  metric, target_value, target_date, effective_from, effective_to
from goals
where current_date >= effective_from
  and (effective_to is null or current_date < effective_to)
order by metric, effective_from desc;
```

### `ingest_events`

Raw payload archive. Lets us replay if the parser changes, and debug HAE's behavior.

```sql
create table ingest_events (
  id            bigserial primary key,
  received_at   timestamptz not null default now(),
  source_ip     inet,
  bytes         integer not null,
  status        text not null,                   -- 'parsed', 'error'
  error         text,
  payload       jsonb not null
);

create index on ingest_events (received_at desc);
```

Retention: keep raw payloads for 30 days, then `payload` is set to `null` (status row stays for audit).

## Migrations

- Each schema change is a separate, sequentially-numbered SQL file.
- The migration tool itself is **deferred** (see `CLAUDE.md`); the migration *files* are written by hand in SQL regardless of tool choice, so the work isn't lost when we pick the tool.
- Migration order for v1 (planned files):

```
migrations/
  0001_extensions.sql              -- btree_gist, citext if needed
  0002_enums.sql                   -- metric_source, meal_kind, etc.
  0003_body_composition.sql
  0004_body_measurements.sql
  0005_foods.sql
  0006_meals.sql
  0007_water_intake.sql
  0008_daily_activity.sql

  0010_sleep_sessions.sql
  0011_goals.sql
  0012_ingest_events.sql
  0013_reminders.sql

```

## Common queries (sketches)

These aren't the final SQL — just sketches that show the schema is shaped right.

**Today's calories + protein:**

```sql
select kcal, protein_g
from daily_nutrition
where day = (now() at time zone 'Europe/Lisbon')::date;
```

**Last 7-day weight sparkline:**

```sql
select date_trunc('day', recorded_at at time zone 'Europe/Lisbon')::date as day,
       avg(weight_kg)::numeric(5,2) as weight_kg
from body_composition
where weight_kg is not null
  and recorded_at >= now() - interval '7 days'
group by 1
order by 1;
```

**Current weight goal and remaining:**

```sql
with active as (select target_value from goals_active where metric = 'target_weight_kg')
select
  (select target_value from active) as target_kg,
  (select weight_kg from body_composition where weight_kg is not null
     order by recorded_at desc limit 1) as latest_kg;
```

**Today's water total:**

```sql
select coalesce(sum(amount_ml), 0) as ml
from water_intake
where (recorded_at at time zone 'Europe/Lisbon')::date
    = (now() at time zone 'Europe/Lisbon')::date;
```
